# Manual test — external Ollama connector

## Prerequisites

- LokLM dev build running (`pnpm dev`).
- Local Ollama daemon (`ollama serve`) reachable at http://localhost:11434, OR a remote Ollama proxy URL.
- Pulled models:
  - `ollama pull qwen3:8b`
  - `ollama pull nomic-embed-text`
  - `ollama pull qwen3:0.6b` (for reranker)

## Scenario 1 — Configure and connect

1. Open Settings (gear icon in title bar) → Advanced tab.
2. Note the warning banner. Expand "External Ollama".
3. Base URL is pre-filled `http://localhost:11434`. Edit if you're using a remote proxy. Tab out — the probe runs and shows green check + version.
4. Pick `qwen3:8b` as LLM model, `nomic-embed-text` as embedder model, `qwen3:0.6b` as reranker model.

Expected: each save shows the green ✓ flash. Settings closed and reopened — selections persist.

## Scenario 2 — Switch LLM to Ollama

1. Still in Advanced → LLM source. Flip to `External Ollama`.
2. Close settings; open Chat; ask a question.

Expected:

- Streamed answer arrives via Ollama.
- Chat header pill shows "via Ollama".
- Title bar LLM status dot turns purple (indicating external Ollama active).

## Scenario 3 — Fallback on Ollama outage

1. While the answer is streaming, kill the Ollama daemon (`pkill ollama` on Linux/macOS, or Task Manager → End Task on Windows).
2. Send a new chat question.

Expected: toast appears bottom-right: "Ollama unreachable — used bundled model." The answer is generated by the bundled model. Header pill shows "via Ollama → bundled (fallback)" briefly.

## Scenario 4 — Switch embedder (re-index gate)

1. Restart Ollama. In Advanced → Embedder, change Source from Bundled to External Ollama.
2. Re-index modal appears. Click "Re-index now".

Expected: probe succeeds; backfill runs across all workspaces; modal closes. After completion, search results return as before but using the Ollama embeddings.

**Note:** the bundled embedder is BGE-M3 (1024 dim) and the suggested Ollama embedder `nomic-embed-text` is 768 dim. The current `chunks.embedding` column is hard-typed `vector(1024)` which will reject the 768-dim vectors. This is a known limitation — a future task should either restrict embedder choice to dim-matching models or migrate the embedding column to a flexible-dim type. Until that's resolved, only choose Ollama embedders that produce 1024-dim vectors (e.g. fine-tuned BGE-M3 variants if available).

## Scenario 5 — Configuration error path

1. Settings → External Ollama → Base URL change to `http://localhost:99999` (invalid).
2. Tab out.

Expected: red error appears with retry button. LLM source toggle still works (falls back to bundled).

## Scenario 6 — Profile editing

1. Settings → Profile tab.
2. Change display name; tab out.

Expected: ✓ saved indicator. Re-open settings — name persists.

3. Click "Change…" and pick a PNG/JPEG.

Expected: avatar preview updates in the modal. Close and reopen — avatar persists.

4. Click "Remove" to clear avatar.

Expected: avatar reverts to deterministic-color initials based on display name.

## Scenario 7 — Install-time opt-in gate (packaged build only)

The connector is an installation option: the wizard's options page has an
"Enable the external Ollama connector" checkbox (default **unchecked**, with
at-your-own-risk wording). The choice is written to `loklm-tier.json` as
`ollamaConnector: true|false`.

1. Run the installer wizard, leave the Ollama checkbox unticked, finish the install.
2. Launch LokLM → Settings → Advanced.

Expected: there is **no Ollama subtab at all** (only LLM, Retrieval, Behavior,
Diagnostics) — no flash of it on open either. `loklm-tier.json` in the install
dir contains `"ollamaConnector": false`.

3. Re-run the installer over the same dir, this time tick the checkbox.
4. Relaunch LokLM → Settings → Advanced.

Expected: the Ollama subtab is back with the normal config UI from Scenario 1
(marker now says `true`).

Edge cases:

- Hand-edit `loklm-tier.json` → delete the `ollamaConnector` key entirely →
  relaunch. Expected: connector available (markers from ≤ v0.4.0 installers
  predate the field; missing key reads as enabled so existing setups keep
  working).
- Dev build (`pnpm dev`): no marker exists → connector always available.
- With the connector locked and a vault that already has Ollama sources
  persisted (restored from an opted-in machine): chat must run on the bundled
  model — the main process refuses to build Ollama providers regardless of
  what the settings say, and `ollama:probe` answers `connector-disabled`.
